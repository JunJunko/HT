# Pivot Note — 键盘酒馆 概念原型 v1 → v2

**日期：** 2026-07-11

## 原假设（v1，HTML 文本框）
在浏览器文本框里打字，测试「输入风格 → 啤酒风味」的映射是否清晰可读，以及客人反应是否有情绪反馈。

## v1 试玩反馈
玩家反馈「和预想的差很远」，具体三点：
- **A 核心机制理解错了**：真正的幻想是游戏驻在屏幕角落、静默读取**全局**键盘输入（玩家在别的软件里工作时）。v1 把玩家关进文本框，还要求按「完成酿造」——恰恰违反 GDD 3.1「不强迫玩家切回游戏」。
- **「不连续」**：v1 是离散的「打字→按完成→出结果」，不是「我一直在工作、酒一直在角落连续酿着」。
- **D 出酒反馈没说服力**：因为输入根本不是玩家真实工作的输入，解释再清楚也无法让人觉得「这杯就是我」。介质本身在否定信息。

根因：把「全局输入」误当成可延后的纯技术问题；但**全局输入就是「好玩」本身**。HTML 沙盒永远读不到其他 app 的按键。

## 保留（从 v1 移植到 v2）
- 风味映射的数学（字母/数字/符号比例 + 节奏 → 烘烤/酒体/苦度/香气/发酵/通道）
- 6 种酒 + 实验酒兜底、加权距离匹配（GDD 14.1）
- 「为什么是它」解释生成器
- 3 位客人偏好匹配 + 带情绪对白
- 修正：空格不再计入符号比例（v1 把空格算进符号，导致正常打字总偏「香料」）

## 改变（v2）
- 介质：**PyQt5 桌面置顶小窗 + pynput 系统级全局键盘监听**（HTML 做不到）
- 形态：边框、置顶、半透明的角落窗口（GDD 11.1 陪伴模式）
- 交互：**删除文本框与「完成」按钮**；窗口被动；酿造按真实时间连续推进，输入加速，空闲也能产出基础酒（GDD 5.4）
- 出酒：不抢焦点的 toast + 窗口内最近结果；一杯约 3-5 分钟（接近 GDD 4 分钟目标）
- 隐私：原始字符即时丢弃，仅留聚合值；常驻「采集中/已暂停」状态 + 一键暂停（GDD 17.4）

## 修正后的假设（v2）
如果一个常驻屏幕角落的小窗口，在玩家正常工作（写代码/聊天/写文档）时静默读取全局键盘输入，酒液在后台连续成形，偶尔浮现一杯完成的酒 + 一位客人——这会让人愿意一直把它开着，并产生「我的工作真的酿出了这杯酒」的感觉。

## 提前撞上的真实课题
pynput 是系统级键盘钩子，**杀毒软件可能误报为键盘记录器**。这正是游戏未来要面对的隐私/分发课题（GDD 17.4），现在就以原型形式遇到了。
