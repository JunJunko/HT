## 1. 本轮整合结论

后续讨论中有四组内容应正式进入 GDD：

1. **出酒揭晓与配方发现系统**：把 Demo 中有趣的提醒气泡升级为核心奖励时刻。
2. **配方标签与距离匹配 V2**：明确稀有度、品质、发现状态的关系，并给出可配置的匹配算法。
3. **玩家传播闭环**：让每次稀有出酒自然生成酒卡、回放和可复现挑战。
4. **世界酒谱**：以发现、奇闻和挑战为核心的可选弱联网系统，而不是比拼挂机时长的传统排行榜。

这些内容增强的是同一条产品主线：

```text
现实输入形成酿造指纹
    → 距离算法决定酒款与品质
    → 气泡制造揭晓期待
    → 客人提供情绪反馈
    → 酒谱、线索和配方引导推动复现
    → 酒卡、挑战与世界酒谱推动传播和共同探索
```

本轮不改变以下基础定位：

- 游戏首先是低打扰的桌面陪伴产品；
- 玩家输入决定结果，随机性只做轻微扰动和平局处理；
- 不保存或上传实际输入字符；
- 普通酒不是废卡，低输入也不会失败；
- 联网功能必须可选，不参与也能完成单机主体内容。

## 2. 三个必须分离的结果维度

酒款结果不再只用一个“星级”概括。系统必须分别记录并展示：

| 维度 | 含义 | 示例 | 主要决定因素 |
|---|---|---|---|
| 配方稀有度 | 这款酒在设计上有多难发现 | 普通、高级、史诗、隐藏 | 配方区域大小、门槛数量、条件特殊性 |
| 本杯品质 | 这一次酿得有多贴近配方 | 粗糙、标准、优秀、完美 | 理想区间距离；原料与设备仅小幅修正 |
| 发现状态 | 该玩家是否已收录这款酒 | 首次发现、已收录 | 玩家酒谱记录 |

允许出现：

- 完美品质的普通酒；
- 标准品质的史诗酒；
- 首次发现但品质粗糙的隐藏酒；
- 已收录但首次达到完美品质的高级酒。

气泡演出读取三个维度，但不能把它们混成单一颜色或单一星级。

## 3. 出酒揭晓与配方发现系统

### 3.1 系统定位

提醒气泡不再只是“酿造完成”的 Toast，而是每隔数分钟发生一次的轻量开奖仪式：

> **酿造过程决定结果，气泡负责制造抽卡式揭晓感。**

玩家忽略气泡时，游戏不得停滞。结果自动进入酒架和最近记录；玩家之后可以补看揭晓。

### 3.2 标准揭晓流程

```text
酿造完成
→ 生成气泡征兆（0.5～1.5 秒）
→ 显示 1～2 个模糊线索（1～2 秒）
→ 玩家点击或延迟自动揭晓
→ 展示酒名、稀有度、品质、发现状态
→ 用 2～3 条原因解释结果
→ 客人反应 / 新客人来访 / 奖励结算
→ 酒谱记录并生成可分享素材
```

普通重复酒的完整流程应控制在 1～2 秒；高级以上首次发现可延长至 3～6 秒。所有长演出均可跳过、补看、静音。

### 3.3 稀有度演出语言

| 稀有度 | 气泡征兆 | 揭晓方式 | 环境反馈 |
|---|---|---|---|
| 普通 | 透明、小幅漂浮 | 轻微破裂后直接展示 | 无强制变化 |
| 高级 | 颜色渐变、少量微粒 | 先显风味标签，再显酒瓶 | 附近客人转头 |
| 史诗 | 膨胀、稳定停顿、轮廓显现 | 酒瓶与名称分步出现 | 灯光、音效和客人明显反应 |
| 隐藏 | 倒流、消失、黑雾或不符合常规的异常 | 延迟显示名称或先触发角色事件 | 专属环境异常 |

隐藏配方不应只是“比史诗更金”。它的识别语言是**异常**，不是更高亮度。

### 3.4 距离和品质的视觉化

- 越接近配方中心，气泡越稳定，酒瓶轮廓越清晰；
- 越接近匹配边缘，气泡越晃动，酒液可略显浑浊；
- 普通酒达到完美品质时也应有独特的稳定、清澈表现；
- 演出只能提示结果，不能在结果确定后再次随机改变酒款。

### 3.5 客人参与揭晓

- 普通酒：已有客人走近并饮用；
- 高级酒：门外出现“有人闻到酒香”的提示；
- 史诗酒：可能触发新客人的剪影和入场；
- 隐藏酒：可先由神秘客人拿走，待饮用后才揭示完整酒名。

由此形成统一解锁链：

```text
特殊输入组合 → 稀有酒 → 特殊客人 → 新原料或新控制能力 → 更多配方
```

## 4. 重复酿造与配方掌握

### 4.1 不解锁“完全自动酿造”

现实输入参与酿造是产品核心。若第 10 次产出后可以直接自动复制目标酒，会绕过输入、发现和揭晓系统。

因此重复酿造奖励应命名为**配方掌握 / 配方引导**：

| 里程碑 | 解锁内容 |
|---:|---|
| 第 1 次 | 酒名、基础风味、酒谱条目 |
| 第 3 次 | 主要形成原因 |
| 第 5 次 | 模糊目标区间和偏差方向 |
| 第 10 次 | 配方预设：自动装配原料、实时相似度提示、锁定一个变量 |
| 第 15 次 | 酒瓶外观或客人特殊对白 |
| 首次完美 | 大师标记、专属气泡细节或酒卡边框 |

“锁定一个变量”代表熟练度提供的控制力，例如稳定发酵温度；其余维度仍由本次现实输入决定。

### 4.2 普通酒经营自动化（后续可选）

若后续经营层需要自动供酒，只允许已完全掌握的普通配方成为酒馆常驻酒：

- 不能发现新配方；
- 不能触发史诗或隐藏配方；
- 品质上限低于完美；
- 不播放完整稀有揭晓；
- 只承担普通客人的基础供应和低额收入。

自动供酒服务经营，现实输入服务探索，两者不能互相替代。

### 4.3 重复结果的价值

重复酒至少同时推进其中一项：

- 客人消费与基础收入；
- 配方掌握里程碑；
- 更精确的形成原因；
- 瓶身、气泡或对白收藏；
- 连续重复时积累“酿酒灵感”。

### 4.4 隐性发现保护

连续重复且没有新发现时，增加隐藏的“酿酒灵感”：

- 优先提高邻近未发现配方的容错，但不能跨越硬门槛；
- 多次接近同一配方时，逐步显示线索；
- 新配方出现后部分清空；
- 不显示“再酿 10 次必出史诗”等手游式保底文案。

玩家看到的是世界内提示，例如：“酒馆里开始出现一种陌生香气。”

## 5. 标签体系与距离算法 V2

### 5.1 四层标签结构

不同职责的标签不能全部塞入同一个欧氏空间。

#### A. 原始行为特征（仅内部计算）

- 字母、数字、符号数量和占比；
- 单位时间输入量；
- 平均输入间隔和标准差；
- 连续输入长度；
- 爆发输入比例；
- 停顿时间比例；
- 三个阶段各自的输入结构。

原始事件即时聚合后丢弃，不保存实际字符。

#### B. 连续酿造语义向量（距离计算核心）

MVP 使用 8 个归一化到 `[0, 1]` 的维度：

| 维度 | 含义 |
|---|---|
| `grain_intensity` | 麦芽基础和投入强度 |
| `roast_level` | 烘烤与深色程度 |
| `bitterness` | 苦度倾向 |
| `sweetness` | 甜度与残糖倾向 |
| `ferment_temp` | 冷 / 暖发酵倾向 |
| `ferment_activity` | 发酵活跃度与爆发性 |
| `stability` | 输入和阶段过程的稳定性 |
| `body` | 酒体厚重程度 |

#### C. 原料、环境与剧情门槛（候选池过滤）

- 是否装备特定原料；
- 客人是否在场；
- 本地游戏时段；
- 前一杯或连续酒款序列；
- 关系或剧情进度；
- 是否拥有特定设备。

“深夜”“葡萄已装备”等条件使用门槛，不编码成连续向量维度。

#### D. 玩家可见展示标签（解释与分享）

例如：花香、果香、坚果、焦糖、清爽、醇厚、暖发酵、深色。它们由语义向量和原料派生，不独立决定结果。

### 5.2 配方使用“允许区间 + 理想区间”

每个配方在各维度上定义两层区间，而不是一个精确目标点：

- `allowed_range`：决定这杯酒是否仍属于该配方；
- `ideal_range`：位于允许区间内部，决定这次是否接近最佳表现。

对任一区间 `[lower, upper]`，单维区间外偏差为：

```text
若 x < lower：δ = lower - x
若 lower ≤ x ≤ upper：δ = 0
若 x > upper：δ = x - upper
```

分别计算允许区间距离 `D_allowed` 和理想区间距离 `D_ideal`：

```text
D_range(recipe) = sqrt( Σ(weight_i × δ_i²) / Σ(weight_i) )
```

只有满足硬门槛且 `D_allowed ≤ match_radius` 的高级、史诗和隐藏配方进入命中集合。

拆成两层是为了避免一个常见错误：若宽泛的普通配方只要落在允许区间内距离就为 0，再直接用该距离计算品质，会导致大量普通酒都被判为完美。

### 5.3 候选选择顺序

```text
1. 生成本次 8 维语义向量
2. 根据原料、环境和剧情过滤硬门槛
3. 计算所有剩余配方的 `D_allowed` 和 `D_ideal`
4. 排除超过各自 match_radius 的非普通配方
5. 先按标准化允许距离 `D_allowed / match_radius` 排序，再用 `D_ideal` 区分同处允许区间内的候选
6. 仅在差值小于 ε（建议 0.03）时，依次考虑隐藏状态、未发现状态和轻微随机
7. 若无特殊配方命中，选择距离最近的普通配方作为保底
```

“未发现配方优先”只处理近似平局，不能让明显更远的酒覆盖玩家输入，否则会破坏复现性。

### 5.4 稀有度定义

稀有度不是先随机抽取，而由配方区域和门槛共同形成：

| 稀有度 | 区域特征 | 条件特征 |
|---|---|---|
| 普通 | 维度少、范围宽、半径大 | 无或极少门槛 |
| 高级 | 多个维度协调、范围较窄 | 常需一种原料 |
| 史诗 | 多维精准组合、半径小 | 原料、设备或进度门槛 |
| 隐藏 | 距离要求高 | 特殊时间、序列、客人或剧情门槛 |

固定稀有度与上线后的“全球发现率”并列展示，永不因全球发现率变化而降级。

### 5.5 品质计算

品质与配方稀有度分离。选定配方后使用理想区间距离计算：

```text
relative_quality_distance = D_ideal / quality_radius
```

建议初始档位：

| 相对距离 | 品质 |
|---:|---|
| `0.00～0.20` | 完美 |
| `0.20～0.45` | 优秀 |
| `0.45～0.75` | 标准 |
| `0.75 以上` | 粗糙 |

`quality_radius` 独立于 `match_radius` 配置。前者控制品质梯度，后者控制配方能否命中，不能混用。输入数量、绝对速度和阶段完成度不直接增加品质；原料协同和设备控制只做小幅修正，且不得改变已经选定的酒款。最终阈值需通过真实输入回放数据校准。

### 5.6 玩家个体归一化

不同职业和输入速度不能永久锁死在单一酒类。前 5～10 杯使用全局默认基线，之后逐步混合玩家历史分位：

```text
normalized_feature = 40% × global_normalization
                   + 60% × personal_percentile
```

- 使用滚动窗口，避免永久历史稀释近期变化；
- 输入速度描述的是“相对自己平时更快或更慢”；
- 极端值截断，长按和机械重复折损；
- 挑战码使用目标分位，不传递原玩家的绝对输入速度。

### 5.7 配方配置示例

```yaml
id: nut_stout
name: 夜班坚果世涛
rarity: epic
match_radius: 0.25
quality_radius: 0.18

gates:
  ingredients: [nut]
  minimum_brew_count: 5

targets:
  grain_intensity: { allowed: [0.70, 0.95], ideal: [0.80, 0.88], weight: 1.0 }
  roast_level:     { allowed: [0.72, 1.00], ideal: [0.82, 0.92], weight: 1.4 }
  bitterness:      { allowed: [0.25, 0.55], ideal: [0.35, 0.45], weight: 0.7 }
  sweetness:       { allowed: [0.45, 0.70], ideal: [0.52, 0.62], weight: 0.8 }
  ferment_temp:    { allowed: [0.55, 0.78], ideal: [0.62, 0.70], weight: 0.8 }
  stability:       { allowed: [0.55, 0.90], ideal: [0.68, 0.82], weight: 0.5 }
  body:            { allowed: [0.70, 1.00], ideal: [0.82, 0.92], weight: 1.2 }

display_tags: [深色, 醇厚, 坚果, 暖发酵]
attract_guests: [tired_accountant, mysterious_brewer]
```

## 6. 传播系统

### 6.1 核心传播循环

```text
玩家正常工作
→ 酿出具有个人特征的酒
→ 气泡揭晓稀有结果
→ 生成酒卡或 6 秒回放
→ 玩家分享
→ 朋友通过挑战码尝试复现
→ 得到复现、变体或意外分支
→ 再次分享
```

传播文案的核心不是“我抽到了一张金卡”，而是：“我的日常输入酿出了这个，你能不能复现？”

### 6.2 个性酒卡

每张酒卡包含：

- 经过策划控制的酒名；
- 配方稀有度、本杯品质、首次发现标记；
- 三个主要成因标签；
- 客人的一句反应；
- 酒瓶或气泡主视觉；
- 游戏名称和稳定识别元素；
- 可选挑战码；
- 不包含桌面、软件名、实际输入内容或敏感数据。

命名可以使用“情境前缀 + 风味核心 + 酒类后缀”，但只能基于输入结构进行幽默化表达，不能断言玩家正在使用 Excel、聊天或写代码。

### 6.3 气泡揭晓回放

高级以上结果保存配置数据和动画种子，在游戏内部重演约 6 秒揭晓：

```text
气泡升起 → 异常征兆 → 轮廓或标签 → 破裂揭晓 → 客人反应
```

导出静态图片、GIF 或短视频时只渲染游戏画面，禁止录制玩家桌面。

### 6.4 酿酒挑战码

挑战码包含：

- 目标配方和目标语义区间；
- 必要原料与可公开环境条件；
- 目标玩家分位而非绝对输入速度；
- 已公开线索和刻意隐藏的线索；
- 配方与算法版本。

接收者得到的是挑战，不是免费酒款。结果允许三种分支：

- 高相似度：成功复现；
- 接近：产生已设计的风味变体；
- 偏离或满足额外门槛：得到其他酒或隐藏分支。

### 6.5 社区共同解谜

正式版约 24～30 款酒中，只选 3～5 款隐藏酒采用社区解谜：

- 不同玩家可获得不同线索；
- 分享卡默认隐藏最后一个条件；
- 榜单展示已破解条件数量，不展示精确区间；
- 达到全服发现人数里程碑后逐步公开更多线索。

普通配方不采用此结构，避免所有内容都依赖查攻略。

### 6.6 周期性身份内容

后续可生成“今日酿酒报告”“每周酒馆周报”和长期流派称号。所有文字只基于聚合指标，避免推断玩家具体工作内容。

## 7. 世界酒谱：弱联网发现与成就系统

### 7.1 产品定位

不制作“总按键数”“挂机时长”“打字速度”的传统总榜。在线系统命名为**世界酒谱**，记录谁发现了什么、发生过哪些稀有事件，以及别人如何参与挑战。

它的首要职责是提供新的探索入口，不是提供竞争性数值奖励。

### 7.2 首发建议栏目

| 栏目 | 记录内容 | 玩家后续动作 |
|---|---|---|
| 首次发现 | 隐藏配方首位、前 10 位、里程碑发现者 | 查看线索、尝试挑战 |
| 本周奇闻 | 最罕见或最离谱的特殊事件 | 观看回放、分享、复现 |
| 社区挑战 | 参与最多、变体最多、挑战链最长 | 导入挑战、查看演化路径 |

“极限酿造纪录”可在验证反作弊成本后加入，但不做单一总分。

### 7.3 特殊事件类型

- **精确条件事件**：一周内用七杯完成七种不同配方；
- **距离事件**：配方相对距离小于 0.05，或同时擦过三个配方边缘；
- **序列事件**：连续数杯沿同一风味方向演化；
- **客人事件**：一杯酒同时令多个性格冲突的客人产生高阶反应；
- **全服事件**：全球第 N 杯、某隐藏酒达到第 N 位发现者、共同破解全部线索。

不设“按键最多”“速度最快”等会鼓励刷键或制造生产力压力的事件。

### 7.4 隐藏信息分层

| 可见级别 | 内容 |
|---|---|
| 所有人可见 | 酒名、外观、稀有度、品质、三个结果标签、发现者、达成人数、揭晓回放 |
| 部分公开 | 已确认原料、大致阶段、已破解线索、挑战入口 |
| 永久不公开或延后公开 | 精确区间、中心向量、距离权重、最后触发条件 |

首次发现者可选择匿名、公开昵称或随机酿酒师编号。不得自动公开电脑用户名；Steam 昵称也应经玩家确认。

### 7.5 奖励边界

世界酒谱奖励只提供荣誉和外观：

- 称号、徽章；
- 酒瓶和气泡皮肤；
- 酒馆纪念牌；
- 客人特殊对白；
- 分享卡边框；
- 首次发现者署名。

不奖励更高稀有酒概率、独占强力原料、大量经济资源或影响排名的能力。

### 7.6 隐私、验证和离线降级

联网为显式选择加入。单机主体在无网络、服务器维护或玩家拒绝参与时仍完整可玩。

客户端只提交可复算的酿造摘要：

```json
{
  "brew_id": "uuid",
  "game_version": "0.2.0",
  "recipe_version": "1.0.0",
  "duration_seconds": 240,
  "feature_vector": [0.81, 0.76, 0.35, 0.62, 0.71, 0.66, 0.83, 0.78],
  "ingredient_ids": ["nut"],
  "environment_flags": ["night"],
  "result_recipe_id": "nut_stout",
  "relative_distance": 0.18,
  "quality": "perfect"
}
```

服务器验证：

- 时长、输入量和提交频率是否合理；
- 特征组合是否矛盾；
- 配方版本和算法版本能否复算结果；
- `brew_id` 是否重复；
- 是否出现短时间大量稀有结果。

禁止上传实际字符、原始按键序列、正在使用的软件、窗口标题、文件名和聊天内容。

由于完全防作弊成本过高，世界酒谱定位为娱乐性荣誉系统，不承载现金、高价值道具或强竞技奖励。

## 8. 开发范围与优先级调整

### 8.1 当前概念原型 / 垂直切片 P0

目标仍是验证“真实输入是否能产生有意义的差异”，新增以下最低内容：

- 用现有 6 种酒覆盖至少普通、高级、史诗三个稀有层；
- 额外配置 1 个隐藏结果或使用测试开关模拟隐藏命中；
- 气泡支持普通、高级、史诗、隐藏四种演出状态；
- 结果明确显示稀有度、品质和发现状态；
- 使用 8 维语义向量、区间距离和硬门槛；
- 同一输入回放应稳定落入同一酒类家族；
- 玩家可补看揭晓，不因忽略通知而损失进度。

此阶段不开发导出视频和服务器。

### 8.2 可传播 Demo P1

- 8～10 种酒，至少包含 1 种史诗和 1 种隐藏；
- 一张完成度足够高的酒卡；
- 可在游戏内重播的 6 秒气泡揭晓；
- 本地生成和导入挑战码；
- 1 位会因史诗酒登场的客人；
- 一套重复酿造掌握里程碑；
- 埋点记录：完整观看、打开分享、保存酒卡、复制挑战码、导入挑战、完成挑战。

此阶段可先手动招募测试者交换挑战码，不要求在线账户。

### 8.3 弱联网验证 P2

- 世界首次发现；
- 本周世界奇闻；
- 社区挑战榜；
- 匿名 / 昵称 / 不参与三种身份选择；
- 服务端复算和基础异常检测；
- 离线队列、失败重试和不联网降级；
- 约 12 个特殊事件：3 个距离、3 个序列、2 个客人、2 个隐藏、2 个挑战事件。

### 8.4 正式版后续 P3

- 配方演化树；
- 周期共同主题；
- 社区共同解谜；
- 每周人格报告；
- 更多极限纪录；
- 经验证后的 GIF / 短视频导出。

不在 P0/P1 同时开发十连抽、卡池轮换、角色星级、碎片升星、赛季通行证或高压力限时活动。

## 9. 新增验收标准

### 9.1 算法

1. 相同特征向量、配置版本和随机种子必须得到相同酒款；
2. 轻微扰动不得跨越距离明显更远的配方；
3. 不满足硬门槛时，隐藏配方绝不进入候选池；
4. 普通酒覆盖所有未命中特殊配方的输入，不能出现失败或空结果；
5. 普通完美和史诗标准必须都能通过测试数据产生；
6. 不同速度玩家经过校准后都能覆盖至少三个酒类家族；
7. 挑战码在不同玩家处按个人分位换算，不比较绝对按键速度。

### 9.2 揭晓体验

1. 普通重复揭晓不超过 2 秒；
2. 稀有首次揭晓不超过 6 秒且可跳过；
3. 忽略通知后结果仍被保存，可稍后补看；
4. 四种稀有度在无文字时也能被测试者区分；
5. 玩家能正确说出“为什么得到这杯酒”的至少一个主要原因；
6. 气泡动画不抢焦点，不遮挡主要工作区域，可完全关闭。

### 9.3 传播验证

1. 酒卡不包含桌面或实际输入内容；
2. 测试者看到酒卡后能识别游戏名称和核心机制；
3. 挑战码可在相同版本稳定还原目标和公开线索；
4. 记录真实的保存、复制、导入和完成行为，不只收集口头“愿意分享”；
5. 目标阈值需在封闭测试前单独设定，不在没有基线时虚构成功率。

### 9.4 世界酒谱

1. 玩家未选择加入前不上传任何酿造摘要；
2. 榜单记录可匿名，且支持退出后停止后续提交；
3. 服务端无法通过上传字段恢复实际输入文本；
4. 榜单不可用时不阻断酿造、图鉴、客人和本地挑战；
5. 从榜单记录发起挑战的转化率应作为核心指标，而不只统计榜单浏览量。

## 10. 与 V0.1 的冲突处理

| V0.1 表述 | V0.2 处理 |
|---|---|
| MVP 明确不联网 | 保持。联网移到 P2 弱联网验证，正式版可选加入 |
| 结果主要使用 1～5 星品质 | 改为稀有度、品质、发现状态三维；星级可作为品质的视觉形式 |
| 固定配方 | 改名为配方预设 / 配方引导，避免理解成自动生产 |
| 配方中心 + 容许区间 | 统一为允许区间 + 理想区间 + 加权距离；命中与品质使用不同半径 |
| 随机性用于相同距离候选 | 保留，但只在标准化距离差小于 `ε` 时使用 |
| 酒谱收集 | 扩展为个人酒谱；联网后增加世界酒谱，二者数据和权限分开 |
| 完成 Toast | 升级为可忽略、可补看的出酒揭晓器 |
| 0～100 满意度与“勉强喝完” | 改为后台风味匹配 + 正向反应层级；不显示分数、不强迫具名客人喝不喜欢的酒 |
| 客人等待两杯后接受相近酒 | 删除等待惩罚；无匹配酒进入酒架或由普通路人消费 |
| 输入提高阶段完成度和品质上限 | 删除；阶段按现实时间推进，输入创造差异而不直接提高品质 |
| 低输入只能获得 1～2 星 | 删除；低输入回归基础向量，仍可能得到高品质普通酒 |
| V0.1 的 8 维结果向量 | 由 V0.2 连续语义向量取代，风味类词汇改为派生展示标签 |

## 11. 仍需工作室确认的决策

1. 稀有度中文最终命名使用“普通 / 高级 / 史诗 / 隐藏”，还是更符合酒馆世界观的名称；
2. 品质使用四档文字、五颗星，还是两者并存；
3. 隐藏气泡的美术语言偏奇幻、都市夜色还是轻微怪诞；
4. 酒名中的职场幽默强度，以及是否提供“中性命名”设置；
5. 挑战码首发仅文本复制，还是同时支持二维码；
6. 世界酒谱是否作为 1.0 首发功能，或在单机主体上线后更新；
7. 在线身份是否依赖 Steam 身份，还是只使用游戏内随机编号；
8. 世界首次发现是否永久署名，以及昵称变更和违规名称的处理规则；
9. 哪 3～5 款隐藏酒适合社区共同解谜；
10. 分享导出的首要格式：静态酒卡、GIF 或竖屏短视频。

## 12. 本轮最终判断

聊天中最值得整合的不是“增加一个抽卡系统”和“增加一个排行榜”这两个孤立功能，而是把现有核心重新组织成一条连贯的发现链：

> **玩家输入决定酿造，距离算法保证可解释与可复现，气泡把结果变成值得期待的时刻，客人让结果产生情绪意义，酒卡与挑战让发现能够传播，世界酒谱让其他玩家的发现反过来成为新的探索目标。**

这条链可以增强产品辨识度，但实现顺序必须是：

```text
先证明输入与出酒有趣
→ 再证明气泡揭晓值得等待
→ 再证明玩家真的会保存和交换结果
→ 最后建设在线世界酒谱
```

如果前一层没有通过真实测试，不应靠增加卡池数量、复杂榜单或运营系统补救。
