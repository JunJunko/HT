# 《键盘酒馆》GDD V0.2 冲突审核与覆盖规则

> 审核对象：GDD V0.1、V0.2 增补稿及前期产品讨论  
> 审核原则：发生冲突时，以“低打扰桌面陪伴、输入决定但不奖励刷键、偶然发现后可解释可复现、客人提供正向情绪反馈、传播和弱联网服务探索”为最高优先级  
> 版本日期：2026-07-20

---

## 1. 文档优先级

在主 GDD 完成一次性合并前，工作室按以下优先级执行：

```text
本冲突审核中的覆盖规则
    > V0.2 增补稿
    > V0.1 基础文档
```

V0.1 保留产品基础和历史决策，但其中被本文件明确列出的规则不再用于开发或验收。

---

## 2. 满意度系统审核

### 2.1 结论

V0.1 的“0～100 满意度 + 40 分以下勉强喝完”与后续聊天的整体方向存在冲突，不能原样保留。

冲突不在于“客人有偏好”，而在于它把客人变成了给玩家打分的裁判：

- 低分和“勉强喝完”会把一次自然酿造解释成失败；
- 玩家可能为了避免负面反应而主动刷键或频繁切回游戏；
- 等待客人连续两杯不匹配，会产生隐性订单压力；
- 史诗或隐藏酒可能很稀有，却因为当前客人不喜欢而得到负面反馈；
- 一个通用数值无法同时表达“是否喜欢这个风味”“这一杯品质如何”“是否触发角色事件”。

### 2.2 替代结构：匹配度与反应层级

客人系统拆成三个概念：

| 概念 | 是否展示数值 | 作用 |
|---|---:|---|
| 风味匹配度 `affinity` | 否 | 在后台决定哪位客人最适合喝这杯酒 |
| 反应层级 `reaction_tier` | 展示文字和动画 | 向玩家提供情绪反馈 |
| 特殊事件条件 `event_gates` | 只显示部分线索 | 决定礼物、剧情、新客人或隐藏事件 |

内部风味匹配可以继续使用客人偏好向量：

```text
affinity = weighted_similarity(beer_tags, guest_preferences)
```

品质只作为小幅加成或特殊事件门槛，不覆盖风味偏好。喜欢清爽酒的客人不会因为一杯完美世涛就突然改变喜好。

### 2.3 玩家可见反应

不再显示 0～100 分，也不使用“差评、嫌弃、勉强喝完”等常规反馈。

| 反应层级 | 含义 | 常规反馈 |
|---|---|---|
| 随手一杯 | 风味匹配较低，但酒仍有日常价值 | 普通路人消费或进入酒架，无负面对白 |
| 合口 | 匹配客人的大类偏好 | 正常消费、简短正向对白 |
| 很喜欢 | 多个偏好标签吻合 | 明显表情、关系进度、较完整对白 |
| 命中灵魂 | 高匹配且满足品质或情境条件 | 礼物、剧情、配方线索或特殊演出 |

这里的反应层级不是另一套品质评级：

- “完美”描述酒本身贴近配方；
- “很喜欢”描述这位客人与这杯酒的关系；
- 同一杯酒可以被不同客人以不同方式喜欢。

### 2.4 消费分配规则

一杯酒完成后按以下方式分配：

1. 在当前客人中寻找风味匹配度最高者；
2. 达到其大类接受阈值时，由该客人饮用并播放对应反应；
3. 无人匹配时，酒进入酒架、成为酒馆常驻散卖或由普通路人消费；
4. 不强迫具名客人喝不喜欢的酒；
5. 不产生关系倒退、金币惩罚或失败提示。

### 2.5 普通客人与稀有客人的区别

- 普通客人接受一类宽泛标签，例如“清爽、淡色、低苦”，品质只影响额外对白；
- 稀有客人的**到店或特殊剧情**可以要求标签组合和较高品质；
- 稀有客人到店后仍应接受一个合理的风味家族，不能永远只等唯一精确配方；
- 精确配方要求只用于一次性剧情、隐藏成就或“命中灵魂”反应，不用于日常消费。

### 2.6 等待规则

删除“连续两杯不匹配后，客人勉强接受相近酒”。改为：

- 客人不会因等待而生气、离店或降低关系；
- 客人正常待机，不形成倒计时订单；
- 玩家可主动查看一条模糊偏好线索，但不会被要求立刻满足；
- 若长时间没有匹配酒，客人可以进行环境互动或点一杯普通常驻酒，不消耗玩家本次探索结果。

## 3. 输入、时长与品质冲突

### 3.1 酿造时长

V0.1 同时写了“每杯固定 4 分钟、三个阶段各 80 秒”和“每阶段自动提供 25% 基础进度、输入提升完成度”，两套逻辑彼此冲突。

覆盖规则：

- 单杯按现实时间连续推进，目标约 4 分钟；
- 三阶段按时间切换，不因玩家是否输入而停止；
- 输入决定酿造特征，不直接缩短倒计时；
- 可允许很小的设备型时长修正，但打字量本身不提高单位时间产量。

这样不会让高频输入玩家获得更多“抽卡次数”，也更符合桌面陪伴的稳定节奏。

### 3.2 低输入处理

删除以下旧规则：

- “输入增加品质上限”；
- “低输入只能得到 1～2 星”；
- “阶段完成度直接增加品质分”；
- “输入越多，处理完成度越高”。

替代规则：

- 无输入或低输入时，特征向量逐渐回归酒馆基础基线；
- 仍会得到普通、稳定的基础酒；
- 如果基础酒的向量恰好落在理想区间，也可以获得优秀或完美品质；
- 活跃输入的价值是带来更丰富的向量变化和更多配方路径，而不是天然更高品质；
- 极少数据时降低的是“风味变化幅度”，不是给酒扣分。

### 3.3 品质唯一主规则

品质以 V0.2 的 `D_ideal / quality_radius` 为主规则。

原料协同和设备控制只做小幅、可解释修正；输入数量、绝对速度和阶段完成度不直接加分。随机波动只用于品质边界附近，且不能跨越多个品质档。

## 4. 配方、稀有度与发现冲突

### 4.1 两套 8 维向量

V0.1 的向量为：

```text
[颜色, 酒体, 苦度, 香气, 果香, 坚果, 香料, 发酵类型]
```

V0.2 的向量为：

```text
[麦芽强度, 烘烤度, 苦度, 甜度, 发酵温度, 发酵活跃度, 稳定度, 酒体]
```

两者不能同时作为配方匹配空间。覆盖规则：使用 V0.2 连续语义向量；颜色、香气、果香、坚果和香料作为派生展示标签或原料结果，不再作为同层距离维度重复计算。

### 4.2 配方命中与品质

使用两层区间：

- `allowed_range + match_radius` 决定是否命中酒款；
- `ideal_range + quality_radius` 决定本杯品质。

不得继续使用 V0.1 的单一相似度同时决定酒款和品质。

### 4.3 稀有度

稀有度不通过独立抽卡概率产生。配方区域越小、协调维度越多、硬门槛越特殊，固定稀有度越高。

随机只处理标准化距离近似平局，不能让明显更远的史诗酒为了制造惊喜覆盖更符合输入的普通酒。

### 4.4 首次发现信息

V0.1 的“首次发现后只从问号变为轮廓”反馈过弱，与气泡揭晓和分享需求冲突。

覆盖规则：首次发现立即显示酒名、稀有度、品质、基础风味和 2 条主要成因；重复酿造再逐步解锁精确区间、偏差方向和配方预设。隐藏酒可以在专属事件演出中延迟数秒显示名称，但事件结束后必须正常收录。

### 4.5 配方固定

“固定配方”统一改名为“配方预设 / 配方引导”。它可以自动装备原料、显示偏差和锁定一个变量，但不自动完成探索酿造。

## 5. 揭晓气泡与低打扰冲突

抽卡式演出不能要求玩家每四分钟切回游戏。

覆盖规则：

- 气泡不抢焦点，不使用系统阻断弹窗；
- 玩家点击时播放完整揭晓，不点击则安静收纳结果；
- 客人消费、解锁和奖励在后台正常结算；
- 稀有结果保留“待补看”标记，玩家展开酒馆后可重播；
- 跳过演出不降低奖励；
- 普通重复酒默认使用极短反馈；
- 静音、关闭粒子、关闭桌面通知均不影响玩法。

因此“抽卡感”是可选择观看的奖励演出，不是新的打扰源。

## 6. 传播系统与隐私冲突

传播内容只能基于游戏已经持有的聚合结果：

- 可以说“数字密集、节奏爆发、暖发酵”；
- 不可以说“你刚才在做报表、和同事聊天、写某种代码”；
- 可以生成幽默的虚构酒名；
- 必须提供中性命名或重新生成选项；
- 内部回放使用结果配置和动画种子，不能录制桌面；
- 挑战码存目标分位和语义向量，不存原始按键序列或绝对 WPM。

社交传播不能成为隐私采集的理由。

---

## 7. 世界酒谱与单机定位冲突

世界酒谱与“Steam 单机桌面陪伴”可以共存，前提是它是可选的发现层：

- 默认本地完成全部酿造、酒谱、客人和挑战；
- 玩家明确选择加入后才上传聚合摘要；
- 不参与联网不会缺少配方、角色或控制设备；
- 榜单不可用不阻断任何本地循环；
- 世界成就只奖励称号和外观，不提高配方概率；
- “全球首次”以服务器首次验证成功的提交为准，离线结果不能事后追溯抢占首次名额；
- 固定稀有度与全球发现率分开，不能根据全服数据动态降级玩家收藏。

传统排行榜中的以下项目永久否决：

- 总输入次数；
- 最长挂机时间；
- 绝对打字速度；
- 通过宏或持续刷键即可领先的总分。

## 8. 内容与范围冲突

### 当前原型

先用少量酒验证算法和气泡，不为凑齐四个稀有度制作大量正式美术。隐藏结果可以先用测试配置验证异常演出。

### 可传播 Demo

建议 8～10 种酒、1 个完整隐藏分支、1 位因稀有酒登场的客人、一套酒卡和本地挑战码。此阶段不建设世界酒谱服务器。

### 弱联网验证

只有在测试者真实保存酒卡、交换挑战码并尝试复现后，才建设首次发现、本周奇闻和社区挑战。

十连抽、卡池轮换、碎片升星、赛季通行证和高压力限时活动与当前方向冲突，不纳入路线图。

## 9. 审核后统一核心循环

```text
现实时间推动一杯酒经过三个阶段
→ 玩家自然输入形成连续语义向量
→ 硬门槛过滤候选，距离算法决定酒款
→ 理想区间距离决定本杯品质
→ 气泡提供可忽略、可补看的揭晓
→ 系统选择最匹配的客人并播放正向反应
→ 重复酒推进掌握，新酒推进图鉴和解锁
→ 玩家可生成酒卡、回放和挑战码
→ 可选世界酒谱把他人的记录变成探索入口
```

这条循环中没有“客人给低分”“输入越多品质越高”“为了榜单刷键”或“必须联网才能发现内容”的环节。

---

## 10. 必须同步修改的实现与配置

后续实现不得继续把以下旧字段作为最终规则：

- `satisfaction: 0..100` 直接驱动正负评价；
- `quality_expectation` 作为普通客人的拒绝条件；
- `completion` 直接加入品质分；
- 单一 `similarity` 同时决定酒款与品质；
- `TargetRecipeId` 被解释为自动生产目标；
- 低输入固定品质上限；
- 未点击 Toast 就不结算结果。

推荐替代字段：

```text
GuestAffinity          // 内部风味匹配
ReactionTier           // 玩家可见反应层级
EventGates             // 礼物、剧情和特殊反应条件
AllowedDistance        // 配方命中距离
IdealDistance          // 品质距离
RecipePresetId         // 引导目标，不是自动生产
RevealState            // unseen / auto_stored / viewed / replayed
OnlineSubmissionState  // local_only / queued / verified / rejected
```

## 11. 最终审核结论

满意度是确认存在冲突的一项，但不是唯一一项。真正需要统一的是游戏对玩家行为的态度：

> 玩家自然输入负责创造差异，不接受生产力考核；客人负责回应与陪伴，不负责判定失败；气泡负责制造惊喜，不负责制造打扰；世界酒谱负责提供线索和故事，不负责鼓励刷榜。

后续所有新增系统都应通过这四条判断。若某项设计要求玩家增加无意义输入、害怕错过通知、承受客人负面评价或为了核心内容被迫联网，则与当前聊天确定的整体方向冲突。
